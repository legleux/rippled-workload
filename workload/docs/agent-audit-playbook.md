# Agent Audit Playbook

How to reproduce the code audit reports using Claude Code agents. The reports in this directory were generated by launching specialized Explore agents with targeted prompts.

## Code Smell / Architecture Audit

**Agent type**: `Explore` (subagent_type)
**Model**: `sonnet` (cheaper, fast enough for read-only analysis)
**Parallelism**: Launch 3 agents simultaneously, one per major file

### Prompts used

Each agent gets a prompt like this (customize the file path and focus areas):

```
Perform a thorough code smell and improvement audit of `workload/src/workload/{FILE}`.

Focus on:
1. **Size/complexity**: How many lines? What distinct responsibilities does it handle? Which should be extracted?
2. **Dead code**: Unused imports, commented-out blocks, unreachable code, debug artifacts
3. **Code smells**: God functions, deep nesting, inconsistent patterns, magic numbers, mutable global state
4. **Modularity**: What could be extracted into separate modules?
5. **Logic issues**: Race conditions, error handling gaps, silent failures
6. **Python 3.13+ opportunities**: Old patterns that could use modern Python

Read the ENTIRE file carefully. Report every issue with line numbers.
```

### Files to audit

| Agent | File | Special focus |
|-------|------|--------------|
| 1 | `app.py` | Dashboard HTML size, mutable globals, lifespan complexity, subprocess blocking |
| 2 | `workload_core.py` | God class (Workload), store confusion, sequence management, init_participants |
| 3 | `txn_factory/builder.py` | Dead builders, cruft from iterations, broken MPToken/NFT builders, TxnContext bloat |

### Supporting files to include

Also audit (can be folded into agent 2 or run separately):
- `sqlite_store.py` — lock contention, connection-per-operation pattern
- `ws.py` + `ws_processor.py` — validation path, event handling
- `constants.py` — completeness
- `fee_info.py`, `randoms.py`, `utils.py` — dead code check

## Transaction Lifecycle Audit

**Agent type**: `Explore`
**Parallelism**: 1 agent (needs cross-file view)

### Prompt

```
Audit the "Transaction Submission Lifecycle" subsystem. Map the full lifecycle,
identify where logic is duplicated or awkwardly placed, and recommend specific
cleanup/modularization steps.

Files to examine:
1. workload/src/workload/app.py — continuous_workload(), submission endpoints
2. workload/src/workload/workload_core.py — Workload class (alloc_seq, submit_pending,
   record_*, _cascade_expire_account, check_finality)
3. workload/src/workload/ws_processor.py — WS validation path
4. workload/src/workload/sqlite_store.py — persistent store
5. workload/src/workload/constants.py — TxState enum

Report:
1. Lifecycle map: every state transition with file:line
2. Logic duplication: same concern in multiple places
3. Coupling issues: what app.py logic should be in workload_core.py
4. Naming confusion: unclear method names, overlapping store semantics
5. Specific recommendations ordered by impact
```

## Output

Reports are saved to `workload/docs/`:
- `code-audit-report.md` — the combined code smell report with prioritized action items (P0-P4)
- `lifecycle-audit-report.md` — the transaction lifecycle architecture audit

## When to re-run

Re-run after:
- Major refactoring (e.g., extracting SequenceManager, TransactionStateMachine)
- Significant feature additions (new transaction types, new stores)
- Before a release or repo split
- When the existing reports feel stale (they reference specific line numbers which shift with edits)

## Tips

- Use `sonnet` model for audits — cheaper and fast enough for read-only exploration
- Launch all 3 file audits in parallel (single message with 3 Task tool calls)
- The lifecycle audit should run separately since it needs cross-file context
- Give each agent explicit instructions to "Read the ENTIRE file" — otherwise they may skim
- Ask for line numbers in every finding — makes the report actionable
